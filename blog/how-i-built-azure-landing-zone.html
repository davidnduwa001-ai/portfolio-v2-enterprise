Step 1 Create Resource Groups (+ tags)

What it is (RG)

A Resource Group is a logical folder for Azure resources; deploy, tag, view costs, and delete as a unit.

Why it matters

AZ-900 loves governance basics: grouping, RBAC scope, and clean teardown.

Clicks

  • Portal → Resource groups → Create
  • Subscription: your sub
  • Resource group: rg-az900-app-<slug>
  • Region: your region (e.g., East US)
  • Tags: Project = az900-landing
  • Review + create → Create

Optional: create a second RG now rg-az900-net-<slug> for networking later.

Step 2 Create an App Service (Web App) on the Free tier

What it is

App Service is PaaS for websites/APIs managed runtime, scaling, no VM patching.

Why it matters

Demonstrates PaaS vs IaaS core AZ-900 topic.

Clicks

  • Create a resource → Web App → Create
  • Basics: Subscription (your sub), Resource Group rg-az900-app-<slug>, Name app-<slug>
  • Publish: Code · Runtime stack: .NET 8 / Node 20 / Python 3.11 · OS: Linux · Region: same as RG
  • Plan: Create new asp-<slug> → Pricing: Free (F1) (if available)
  • Monitoring: App Insights can be Off for now
  • Review + create → Create

Verify

After deploy: Web App → Overview → click Default domain https://app-<slug>.azurewebsites.net — you should see a splash/runtime page.

Step 3 Put simple content on it

Why: Proves deployment basics; AZ-900 expects you understand content paths.

Option A External Git

Web App → Deployment Center → External Git → point to a tiny public repo.

Option B ZIP deploy

Web App → Deployment Center → ZIP deploy → upload a zip with index.html (“Hello from Azure App Service”).

Refresh your site URL to see content live.

Step 4 (Optional) Add Azure Front Door

What: Global edge entry, caching, acceleration; optional WAF. Why: Shows edge & global distribution concepts.

If you want $0 today, skip and add later.

Clicks

  • Create a resource → Front Door and CDN profiles → Create
  • RG: rg-az900-app-<slug> · Name: afd-<slug> · SKU: Standard (Front Door)
  • Endpoint: fd-<slug> → Origin type: App Service → select your app-<slug>
  • Route: pattern /*, HTTPS redirect On

Open https://fd-<slug>.azurefd.net and confirm the site.

Step 5 Cost guardrails: Budget + email alerts

Why: Cost visibility & governance = AZ-900 gold.

Clicks

  • Portal search → Cost Management + Billing → Cost Management → Budgets → Add
  • Scope: Subscription (or your RG)
  • Reset period: Monthly · Amount: e.g., $5
  • Filters: Tags → Project = az900-landing (if you tagged RG/resources)
  • Alerts: add your email at 80% and 100% → Create

Verify

Cost analysis → filter by Tag Project = az900-landing → see current spend (likely $0 on Free tier).

Step 6 Simple availability alert (HTTP 5xx)

What it is

Alerts notify you when metrics/logs breach a threshold.

Why it matters

AZ-900 expects basic monitoring vocabulary.

Clicks

  • Monitor → Alerts → Create → Alert rule
  • Scope: select your Web App (App Service)
  • Condition: Signal type Metrics → Metric Requests → Add filter → set HTTP status code group = 5xx
  • Aggregation: Total · Operator: Greater than · Threshold: 5
  • Period: 5 minutes · Evaluation frequency: 1 minute
  • Action group: Create new → add your email (SMS optional)
  • Review + create → Create

Step 7 (Optional) Landing-zone “skeleton” network

What: VNet with subnets + NSGs to demonstrate segmentation. Why: Speak to subnets/NSGs; later, secure PaaS with Private Endpoints.

Clicks

  • Create a resource → Virtual network → Create
  • RG: rg-az900-net-<slug> · Address space: 10.0.0.0/16
  • Subnets: snet-web 10.0.1.0/24, snet-data 10.0.2.0/24
  • Create two NSGs and associate one to each subnet; keep inbound locked down by default.

Lessons learned

  • Tags make cost analysis painless and professional.
  • App Service is the quickest PaaS landing for demos & labs.
  • Start on Free, then scale or add Front Door when needed.
